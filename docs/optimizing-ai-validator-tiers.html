Optimizing AI: Cheap Validator Tiers

Use tools created by AI to reduce AI waste, improve agent throughput, and make the next generation of tools cheaper to operate.

Thesis

Most validator work should not spend scarce Spark/Codex budget. If builders emit structured evidence, validation becomes a small decision problem:

  1. Check deterministic facts.
  2. Ask a cheap model only when judgment is needed.
  3. Escalate to Spark/Codex only when the validator must reason across code, diagnose failures, or produce a fix.

Validator Routing

builder output
  -> validation packet
  -> deterministic gate
  -> cheap model judge
  -> Spark/Codex only for complex or failed cases

Tier 0: No Model

Command exit codes, file existence, JSON schema, API smoke checks, screenshot dimensions, and log pass/fail markers.

Tier 1: Cheap Judge

Read a compact validation packet and return approve, needs_fix, or blocked.

Tier 2: Spark/Codex

Inspect code, debug ambiguous failures, write repairs, or validate high-risk architectural changes.

Required Validation Packet

{
  "schema": "cento.validation-packet.v1",
  "task": "#123",
  "claim": "Review page shows structured summary above logs.",
  "risk": "low",
  "checks": [
    {
      "name": "syntax",
      "status": "passed",
      "evidence": "node --check templates/agent-work-app/app.js"
    },
    {
      "name": "api",
      "status": "passed",
      "evidence": "GET /api/review/123 includes review_summary"
    },
    {
      "name": "screenshot",
      "status": "passed",
      "evidence": "workspace/runs/agent-work/123/screenshots/review.png"
    }
  ],
  "decision_requested": "approve"
}

Cheap Judge Prompt

You are a strict Cento validation judge.
Return only JSON with:
  decision: approve | needs_fix | blocked
  reason: one short sentence
  missing_evidence: string[]

Rules:
- Approve only if all required checks passed and evidence paths are present.
- Use needs_fix if implementation evidence exists but a check failed.
- Use blocked if external access, credentials, devices, or human decisions are required.
- Do not infer from missing data.
{
  "decision": "approve",
  "reason": "All required checks passed and screenshot evidence is present.",
  "missing_evidence": []
}

Operating Policy

  • Default simple validators to Tier 0.
  • Do not introduce a Tier 1 model judge until deterministic validation has real usage data.
  • Reserve Codex for implementing the validator and for explicit human-requested follow-up, not automatic escalation.
  • Track estimated AI cost per issue, even when the value is zero.
  • Prefer a plain JSON or Markdown run summary over new dashboard UI in the first hour.

Roadmap

This is a one-hour E2E implementation plan, not a corporate program. The goal is a thin but working validator loop that can prove value before any larger spend.

Slice Timebox Build Done When
0. Scope Lock 5 min Use one schema, one output path, and three deterministic check types: file exists, command exit code, and JSON field present. No cheap model integration, no general dashboard redesign, and no broad policy engine in the first pass.
1. Packet Writer 15 min Add a small helper that writes validation-packet.json with task, claim, risk, checks, decision requested, and timestamp. A sample run produces a valid packet in the task workspace without requiring model calls.
2. Tier 0 Gate 20 min Add deterministic evaluation that marks each check passed, failed, or missing and decides approve only when every required check passes. Failed or missing evidence returns needs_fix or blocked without spending AI budget.
3. Routing Stub 10 min Add explicit routing fields: validator_tier, ai_calls_used, estimated_ai_cost, and escalation_reason. Tier 0 approvals show zero AI cost; failures include a clear reason for human/Codex follow-up.
4. Smoke Test 10 min Run one passing packet and one failing packet, then record the outputs in the task folder or existing run log. The E2E path demonstrates approval, failure, and zero-cost accounting in under one hour total.
Defer model judging, batching, historical dashboards, and advanced escalation packaging until Tier 0 has proven that it saves real repeated work. The first implementation should be boring and local.

AI Budget

Hard cap: one Codex hour to implement the first E2E version. Runtime AI budget for Tier 0 is zero.

Item Budget What It Buys What It Does Not Buy
Codex implementation Max 60 min One helper, one deterministic gate, one sample passing packet, one sample failing packet, and clear output files. No generalized workflow engine, no new service, no full Taskstream UI rewrite.
Runtime AI $0 for Tier 0 Approvals and failures based on deterministic evidence only. No semantic judgment and no reading source code to infer intent.
Optional Tier 1 Deferred Later cheap-model review for ambiguous packets after Tier 0 is useful. Not part of the first hour; do not spend model budget until there are real ambiguous packets.
Optional Tier 2 Manual only A failed packet can tell Codex what evidence is missing. No automatic Spark/Codex escalation pipeline in the first version.
Reporting 5 min inside the hour Append or emit a small JSON/Markdown summary with tier, decision, and estimated AI cost. No charting, no monthly analytics, no dashboard redesign.
Success metric for the first hour: one low-risk task can be validated end to end with zero runtime model calls, and a failed task produces an actionable missing-evidence reason.

Potential Caveats

  • False approvals: deterministic checks can still approve the wrong thing if the check is too weak. Mitigation: keep claims narrow and require exact evidence paths or exact JSON fields.
  • Packet gaming: builders could emit optimistic claims. Mitigation: Tier 0 should verify evidence independently instead of trusting claim text.
  • Overbuilding risk: the system can become more expensive than the validation it replaces. Mitigation: one helper, one gate, one output summary in the first version.
  • Context loss: compact packets may hide architectural risk. Mitigation: mark migrations, security-sensitive code, data loss risk, and cross-module contracts as manual review.
  • Fake savings: avoided AI spend is meaningless if humans spend more time debugging the validator. Mitigation: measure whether the packet made the next review faster.
  • Model dependency: Tier 1 should remain optional. Mitigation: the first usable version must work with no model calls after implementation.
