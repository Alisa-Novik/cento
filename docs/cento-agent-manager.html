Cento Agent Manager

Build a managed operations layer that continuously inspects Cento agent activity, detects stale or wasteful work, and provides hardcoded management tools for safe remediation.

Executive Summary

Cento currently relies on several independent signals for agent health: tmux sessions, process trees, run ledgers, issue status, pool reports, logs, and manual operator observation. The recent stale-validator incident showed that these signals can disagree and create confusing UI output.

Cento Agent Manager should become the standard control plane for agent operations. It should scan for stale, idle, stuck, errored, duplicated, and low-value agent work; produce actionable recommendations; and expose a deliberately small toolkit for managing Cento agents.

Business Need

  • Reduce wasted model limits caused by stuck validators, duplicate builders, dead ledgers, and idle shells.
  • Improve operator trust by separating live work from stale historical evidence.
  • Create a repeatable process for agent cleanup, timeout policy, and escalation.
  • Provide clear metrics for agent throughput, failure modes, cost pressure, and review bottlenecks.

Scope

In Scope

  • Read tmux, process, run-ledger, issue, and pool state.
  • Classify agents as live, stale, idle, stuck, errored, duplicate, manual, or unmanaged.
  • Detect suspicious patterns such as empty logs after long runtime, low CPU for active work, repeated failed validators, and issue status mismatches.
  • Generate operator-facing insights with recommended commands and confidence levels.
  • Expose hardcoded management actions through a small MCP-style toolkit and CLI wrapper.

Out of Scope for MVP

  • Fully autonomous killing or dispatching without explicit policy gates.
  • Generic third-party process management outside Cento agent sessions.
  • Cost accounting beyond approximate runtime and model/provider attribution.

MVP Requirements

  1. Create `scripts/agent_manager.py scan --json` to produce a normalized inventory of live processes, tmux sessions, run ledgers, issue state, and pool state.
  2. Create stuck-agent heuristics: no log growth, idle CPU, long elapsed runtime, child process wait, missing tmux session, done issue with active run, and duplicate active runs per issue.
  3. Create `scripts/agent_manager.py report` to print concise operator recommendations and write JSON/Markdown evidence under `workspace/runs/agent-manager/`.
  4. Create explicit actions for mark-stale, mark-blocked, terminate-session, attach-log, reconcile-ledger, and generate-ticket.
  5. Add tests using fixture ledgers for stale validators, stuck Playwright command, duplicate issue runners, and manual shells.
  6. Update the Agent Processes panel to consume manager output when available.

Hardcoded MCP Toolkit

The toolkit should be intentionally narrow. It can be exposed first as local functions and later as an MCP server. Each action must return a structured result with `dry_run`, `evidence`, `command`, and `risk`. 

cento.agent_manager.scan()
cento.agent_manager.classify(run_id=None, issue_id=None)
cento.agent_manager.recommend(limit=20)
cento.agent_manager.mark_stale(run_id, reason)
cento.agent_manager.mark_blocked(issue_id, reason, evidence)
cento.agent_manager.terminate_tmux(session, reason, dry_run=True)
cento.agent_manager.reconcile_ledger(run_id, dry_run=True)
cento.agent_manager.open_evidence(run_id=None, issue_id=None)
cento.agent_manager.create_investigation_ticket(title, description, severity="SEV2")
cento.agent_manager.pool_stats(window="24h")

Implemented CLI

The MVP is implemented as scripts/agent_manager.py. It keeps destructive actions dry-run by default and writes durable reports under workspace/runs/agent-manager/. 

cento agent-manager scan
cento agent-manager scan --json
cento agent-manager report
cento agent-manager recommend --limit 10
cento agent-manager classify --issue-id 81
cento agent-manager mark-stale RUN_ID --reason "stuck validator" --dry-run
cento agent-manager mark-blocked 81 --reason "stuck validator" --evidence RUN_ID --dry-run
cento agent-manager terminate-tmux cento-agent-81-095103 --reason "stuck validator" --dry-run
cento agent-manager reconcile-ledger RUN_ID --apply

reconcile-ledger --apply is intentionally bounded. It archives only stale ledger rows attached to Done issues and leaves stale rows on open issues actionable for requeue, validation, or blocker triage.

Operating Policy

  • Default mode is observe and recommend.
  • Mutating actions require a policy flag, explicit operator action, or a ticket-owned workflow.
  • Any kill action must capture process tree, tmux pane, ledger state, issue state, and log size before termination.
  • Done issues with stale ledgers should not be shown as running work.
  • Historical stale ledgers for Done issues may be archived out of active views; stale ledgers for open issues must remain visible as actionable risk.
  • Long-running validators with empty logs should be escalated automatically as likely stuck.

Acceptance Criteria

  • Manager identifies the current `#81` stuck validator pattern and recommends a safe remediation.
  • Manager distinguishes stale `#65/#77` ledger rows from live agent processes.
  • Manager writes a durable report with enough evidence for a human or AI worker to act without chat context.
  • Manager exposes the hardcoded toolkit actions behind dry-run defaults.
  • Agent Processes panel can display manager-derived live, stale, manual, and risk counts.
