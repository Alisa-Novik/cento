Overview

The tracker is a thin Cento layer over your Linux Redmine stack. Redmine stores the board; Cento provides the command line workflow that agents can follow.

Project cento-agent-work

A dedicated Redmine project for agent issues.

Trackers Epic / Task

Use epics for packages and tasks for agent-sized work.

Status flow Queued -> Done

Every agent should keep the Redmine issue current.

Run bundles workspace/runs

Prompts, dispatch metadata, and Codex logs live in repo-local run folders.

Operating model: create or split work, assign nodes, let agents claim tasks, keep notes in Redmine, and inspect cento cluster activity for the live machine state.

Bootstrap

Run this once from either node. It creates the Redmine project, statuses, trackers, and fields. It is idempotent, so re-running it is safe.

bootstrap
cento agent-work bootstrap

Default transport

On macOS, Cento uses alice@alisapad.local when available, then falls back to the cluster SSH mesh.

force cluster mesh
CENTO_REDMINE_TRANSPORT=cluster cento agent-work list

Override the host

Use this when the Linux host name changes or you want a different SSH route.

custom ssh target
CENTO_REDMINE_SSH=alice@alisapad.local cento agent-work list

Create One Task

Use create for one focused task. It writes the Redmine issue, stores node/owner/package fields, and adds a starting journal note.

create task
cento agent-work create \
  --title "Improve mission control pane" \
  --description "Add a compact running-agent summary and link it to cluster activity." \
  --node linux \
  --agent codex \
  --package mission-control

List active work

active board
cento agent-work list

Inspect a task

issue details
cento agent-work show 123

Split Work Across The Cluster

Use split when one larger request should become a package of agent-sized issues. Node assignment rotates through the list passed to --nodes.

split package
cento agent-work split \
  --title "Mission control MVP" \
  --goal "Track what agents are doing as first-class Redmine work." \
  --nodes linux,macos \
  --task "Create Redmine-backed task model and CLI" \
  --task "Add Mac mission-control tile view" \
  --task "Document agent operating procedure"
Created item Purpose How it is tracked
Agent Epic Represents the package. Grouped by Cento Work Package.
Agent Task Represents one agent-sized unit. Assigned to linux, macos, or another node.
Custom fields Make Redmine useful as a board. Node, owner, state, package, and dispatch metadata.

Agent Lifecycle

Agents should move issues through a short explicit state machine. Review means code is ready for human inspection. Done means verification passed.

Queued

Created, assigned, and waiting for an agent.

Running

Claimed by an agent and actively being worked.

Review

Implementation is ready to inspect.

Blocked

Needs a decision, access, or repair.

Done

Verified and closed.

lifecycle commands
cento agent-work claim 123 --node linux --agent codex
cento agent-work update 123 --status running --note "Found the dashboard entry point."
cento agent-work update 123 --status review --note "Implemented; make check passes."
cento agent-work update 123 --status blocked --note "Blocked waiting for Redmine access."
cento agent-work update 123 --status done --note "Verified and closed."

Agent Lanes

Scaling works when ownership stays separate. Builders implement, Validators verify, Docs/Evidence preserves proof, and Coordinators keep routing, status hygiene, and escalation current.

Builder

Implements the smallest coherent code or content change and hands off with focused evidence.

Validator

Runs independent checks and moves work to Review only after required evidence passes.

Validator contract

Docs/Evidence

Maintains hubs, screenshots, validation logs, and manager-facing review notes.

Docs/Evidence contract

Coordinator

Splits stories, routes workers, plans pools, keeps statuses honest, and escalates blockers early.

Coordinator contract

Give An Agent The Prompt

prompt prints the work packet for an issue: subject, node, owner, package, description, and update protocol.

Print prompt

stdout
cento agent-work prompt 123

Write prompt file

file output
cento agent-work prompt 123 --output workspace/runs/agent-work/issue-123-prompt.md

Dispatch Through Cento Cluster

Dispatch starts a detached tmux session on the target node through cento cluster exec. Dry-run first when you want to verify the node, model, session name, and prompt path.

inspect dispatch
cento agent-work dispatch 123 --node linux --agent codex --dry-run
start agent session
cento agent-work dispatch 123 --node linux --agent codex --model gpt-5.3-codex-spark
live node state
cento cluster activity linux
cento cluster activity --json linux

Redmine is the assignment and lifecycle record. cluster activity is the live execution view: tmux sessions, Codex processes, and recent pane text.

MVP Limits And Verification

The MVP is deliberately practical and local-first. It is intended for this trusted self-hosted Redmine stack.

Limits

Direct database writes are used for now. Dispatch expects codex, tmux, and a Cento checkout on the target node. Package hierarchy is represented by the Cento Work Package field.

Verification

Run the E2E probe after changes to the tracker.

e2e
make agent-work-e2e
