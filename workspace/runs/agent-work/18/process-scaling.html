Cento Dev Process Scaling

Short operating report from the iPhone Cento PWA work. Focus: remove coordination drag, make validation independent, and keep review evidence easy to find.

Next Steps

Executive Take

Split validation from implementation Implementation should not own final screenshots, API evidence, and Redmine proof. A dedicated validation agent should run in parallel and block review if evidence is missing.
Promote deliverables hub to standard Every story should output one start page with links to the app, docs, screenshots, validation log, and Redmine status.
Make story contracts smaller Bundle related UX work, but require a one-line acceptance contract and exact evidence paths before work starts.

Inefficiencies Seen

  1. Validation, screenshots, Redmine updates, and implementation are currently handled by one worker, which creates serial waiting and easy context drift.
  2. Deliverables exist, but discovery depends on chat memory unless the navigation hub is updated every time.
  3. Screenshot capture is manual and repeated; viewport, URL, token, and output paths should be generated from a story manifest.
  4. Redmine comments can become dense. Review notes need the same fixed structure every time: Delivered, Validation, Evidence, Residual risk.
  5. Physical iPhone testing is still not truly automated from this Linux context, so LAN/PWA validation needs a clear human handoff checklist.

Recommended Agent Lanes

Builder Owns code changes only. Produces implementation notes and does focused local tests.
Validator Runs API checks, browser screenshots, mobile viewport review, and reports defects back before Redmine review.
Docs/Evidence Maintains start-here.html, validation logs, screenshots index, and manager-facing summaries.
Coordinator Splits Redmine stories, keeps statuses current, sends notifications, and decides when to combine small items.

Immediate Changes To Make

1. Story manifest

One file per story

Create story.json with routes, API endpoints, screenshots, validation commands, and Redmine id. Agents read the same contract.
2. Screenshot runner

Automate evidence

Add a script that takes the manifest and captures desktop/mobile screenshots with consistent names and viewport sizes.
3. Review gate

Block weak review notes

Do not move a story to Review unless syntax checks, API checks, screenshot paths, and visual inspection notes are present.
4. Hub updater

Keep navigation current

Generate or patch the deliverables hub from story manifests so the manager does not need to ask where files are.
5. iPhone handoff

Make device testing explicit

Each mobile story should include LAN URL, token path, exact taps, expected result, and one screenshot slot for physical device evidence.
6. Notification rule

Notify only on state changes

Send notifications when work starts, needs human input, reaches Review, or fails validation. Avoid noisy progress pings.

Validation Agent Contract

  • Start from the story manifest, not chat history.
  • Run API checks before opening the UI.
  • Capture required screenshots.
  • Inspect screenshots manually and note defects.
  • Return pass/fail plus exact artifact paths.

Docs Agent Contract

  • Update the hub on every story transition.
  • Keep manager pages short and action-first.
  • Link screenshots directly.
  • Keep captain's notes append-only.
  • Preserve old evidence instead of rewriting history.

Coordinator Contract

  • Combine stories when validation evidence is shared.
  • Split stories when screenshots or APIs differ.
  • Keep Redmine statuses accurate.
  • Assign validation before implementation finishes.
  • Escalate missing device access early.

Suggested Next Sprint Shape

  1. Day 1 Add story manifest format and screenshot runner for the PWA gateway.
  2. Day 1 Create a reusable Redmine review-note template with required evidence sections.
  3. Day 2 Add hub generation or a strict hub update command.
  4. Day 2 Run one story with Builder, Validator, and Docs/Evidence lanes separated.
  5. Day 3 Decide whether physical iPhone validation stays manual or needs a Mac/iPhone automation lane.

Resource Ask

Minimum useful upgrade: one validation agent lane plus a screenshot runner. Best upgrade: one always-on Mac/iPhone lane for native install, Safari PWA behavior, and device screenshots. That removes the current Linux-to-iPhone blind spot.
